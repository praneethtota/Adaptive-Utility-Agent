1. The problem this solves

Deployed language models are static. Once released, their behavior is frozen. If a model produces a wrong answer today — a hallucination, a complexity claim that contradicts the code, a medical fact that has since been updated — it will produce the same wrong answer tomorrow, and every day until the next version ships. There is no feedback loop between detected errors and model behavior between releases.

This framework builds that feedback loop. It wraps any LLM with a layer that:

  • Detects errors and contradictions in outputs automatically
  • Stores verified corrections persistently across sessions
  • Injects those corrections into future responses immediately (no retraining required)
  • Accumulates DPO training pairs so that calibration cycles actually fix the model weights
  • Gates responses with a hard abstention rule when confidence falls below a domain-specific floor

The governing mechanism is a utility function that serves as a control law — not a monitoring metric — over all of these behaviors simultaneously. Here is the full system in one diagram:

3. The three phases

The framework is designed to be built incrementally. Phase 1 is useful on its own. Each phase extends it without breaking what came before.

Phase 1 Simulated ✓

Monolithic wrapper

One base model. Utility scoring, contradiction detection, session correction injection, assertions store, calibration-cycle DPO pairs, LoRA fine-tuning pipeline, personality wrapper. No routing. This is what the simulation runs.

Phase 2 Designed

Routing + specialists

Field classifier routes queries to domain specialist models. Arbiter resolves conflicts between specialists. Blue-green deployment per specialist, independent of each other. Same utility function throughout.

Phase 3 Future

VCG arbitration + hardware graph

Utility-governed arbitration via VCG mechanism (truthful reporting is dominant strategy). Hardware-adaptive graph depth — shallow on high-VRAM, deep specialist graph on consumer GPUs. See §10.6 for proofs.

Build Phase 1 completely before touching Phase 2. The correction loop must work correctly on a single model before routing adds complexity. Routing cannot improve what the correction loop cannot fix — it will only route bad outputs to different specialists instead of correcting them.

4. The utility function

Why it exists

Most agent frameworks monitor quality after the fact. This framework uses a utility score as a control law that governs every decision the system makes:

  • At inference time — if U falls below minimum bounds, the agent abstains instead of answering
  • At correction time — contradictions penalize confidence proportional to the field's error cost
  • At training time — the field penalty multiplier is the DPO loss weight, so a wrong answer in surgery is trained against 10× harder than one in creative writing
  • At deployment time — a new LoRA adapter is only promoted if it passes a benchmark gate measured in U

The formula

U = w_e(f) · E_ema  +  w_c(f) · C  +  w_k(f) · K_effective

E_ema  — Efficacy (EMA-accumulated): performance vs human baseline
C      — Confidence: internal consistency, penalized by contradictions
K      — Curiosity: exploration bonus for novel domains (capped at 50% of U)
f      — field, which determines weights and minimum competence thresholds

Decision rule:
  act    if C ≥ C_min(f) AND E_ema ≥ E_min(f)
  abstain + escalate  otherwise

The additive weighted structure is not arbitrary — it is the unique functional form satisfying five behavioral axioms (proved from first principles in Appendix B, Theorem B.1). The weights are set proportional to the cost of each error type in the field, calibrated against professional licensing standards.

Blending configs under ambiguity

When a query spans multiple fields, get_effective_config() blends configs by probability weight, which naturally makes the system more conservative under ambiguity — the high C_min of any present high-stakes field pulls the blended minimum up.

agent/config.py — blending under ambiguity view on GitHub ↗ 
def get_effective_config(field_distribution: Dict[str, float]) -> FieldConfig:
    """
    When field is ambiguous, blend configs by probability weight.
    This makes the agent more conservative under ambiguity.
    """
    for field, prob in field_distribution.items():
        cfg = FIELD_CONFIGS.get(field, FIELD_CONFIGS["general"])
        blended.w_confidence += prob * cfg.w_confidence
        blended.c_min        += prob * cfg.c_min
        blended.penalty_multiplier += prob * cfg.penalty_multiplier
        ...
    return blended

5. The assertions store

Why it exists

Session-level correction injection — adding a note to the system prompt that says "don't claim bubble sort is O(n)" — fixes the current conversation. It does nothing for the next session, or the next user. The assertions store is the cross-session memory layer: a persistent fact store where verified corrections accumulate with field-specific confidence decay.

The critical design choice is the decay model. Not all facts age at the same rate. A mathematical proof (bubble sort is O(n²)) does not decay — it will be true in 100 years. A clinical guideline may be superseded in 18 months. The store encodes this directly:

agent/assertions_store.py — decay classes view on GitHub ↗ 
class DecayClass(str, Enum):
    A = "A"  # no decay — proven mathematical/logical facts
    B = "B"  # slow decay — τ = 10 years (mechanical engineering, classical physics)
    C = "C"  # moderate decay — τ = 3 years (medicine, law, architecture)
    D = "D"  # fast decay — τ = 6 months (clinical guidelines, ML benchmarks, security)

DECAY_TAU_SECONDS = {
    DecayClass.A: None,                    # immortal
    DecayClass.B: 10 * 365.25 * 86400,    # 10 years
    DecayClass.C: 3  * 365.25 * 86400,    # 3 years
    DecayClass.D: 0.5 * 365.25 * 86400,   # 6 months
}

# Default decay class per field — subject overrides take priority
FIELD_DECAY_CLASS = {
    "surgery":                DecayClass.C,
    "structural_engineering": DecayClass.B,   # physics principles very stable
    "software_engineering":   DecayClass.D,   # best practices change fast
    "mathematics":            DecayClass.A,   # proofs don't expire
}

# Subject-level overrides always win over field default
SUBJECT_DECAY_OVERRIDES = {
    "time_complexity":    DecayClass.A,   # algorithmic complexity is Class A
    "algorithm_correctness": DecayClass.A,
    "clinical_guideline": DecayClass.D,
    "ml_benchmark":       DecayClass.D,
    "api_version":        DecayClass.D,
}

Effective confidence over time

agent/assertions_store.py — Assertion.effective_confidence() view on GitHub ↗ 
def effective_confidence(self, now=None) -> float:
    """
    C_effective = C_at_write × exp(-Δt / τ)
    Class A: no decay — C_effective = C_at_write always.
    """
    if self.decay_class == DecayClass.A:
        return self.confidence_at_write   # proofs don't age

    tau = DECAY_TAU_SECONDS[self.decay_class]
    delta_t = (now or time.time()) - self.timestamp
    return self.confidence_at_write * math.exp(-delta_t / tau)

When the store is queried, only assertions whose effective confidence exceeds the threshold are returned. Old clinical guidelines fade out of active use without explicit deletion — the decay handles it automatically. Use store.prune_stale() to garbage collect below a floor (default 0.05).

6. The contradiction detector

Why it exists

The correction loop can only correct what it can detect. The contradiction detector is the layer that turns model outputs into structured error signals. It runs four checks in order of increasing cost, short-circuiting when a contradiction is found:

agent/contradiction_detector.py — the check pipeline view on GitHub ↗ 
def check(self, problem, solution, claimed_complexity=None) -> ContradictionResult:
    result = ContradictionResult()

    self._check_syntax(solution, result)               # 1. Is the code valid Python?
    self._check_logical_consistency(problem, solution, result)  # 2. Fails own test cases?
    if claimed_complexity:
        self._check_complexity_claim(solution, claimed_complexity, result)  # 3. Math wrong?
    self._check_cross_session(problem, solution, result)  # 4. Contradicts prior solutions?

    # Total penalty: sum of severities × multiplier, capped at 0.5
    if result.contradictions:
        total_severity = sum(c.severity for c in result.contradictions)
        result.confidence_penalty = min(
            total_severity * self.penalty_multiplier * 0.1,
            0.5   # never destroy confidence in a single interaction
        )
    return result

How it caught the two_sum contradiction

The mathematical check is what catches the seeded two_sum contradiction. It uses Python's ast module to count nested loop depth, then compares that against the claimed complexity:

agent/contradiction_detector.py — complexity check via AST view on GitHub ↗ 
def _check_complexity_claim(self, solution, claimed_complexity, result):
    code = self._extract_code(solution)
    nested_loop_count = self._count_nested_loops(code)   # uses ast.parse()

    if "o(n)" in claimed_complexity.lower() or "o(1)" in claimed_complexity.lower():
        if nested_loop_count >= 2:
            result.contradictions.append(Contradiction(
                type="mathematical",
                description=f"Claimed {claimed_complexity} but code has "
                            f"{nested_loop_count} nested loop levels",
                severity=0.7
            ))

# For two_sum cycle 1:
# claimed_complexity = "O(n)"  ← from CYCLE_RESPONSES
# nested_loop_count  = 2       ← two for-loops detected by AST
# → Contradiction detected, severity=0.7, penalty applied to confidence

The penalty flows directly into the utility score. contradiction_result.confidence_penalty is passed into utility_scorer.score() as an argument. There, it is multiplied by field_config.penalty_multiplier before being applied to the EMA update. For software_engineering (penalty_multiplier=2.0), a severity-0.7 contradiction produces a 14% confidence reduction. For surgery (penalty_multiplier=10.0), the same contradiction produces a 70% reduction.

7. The confidence updater

Why EMA, not a running average

The confidence signal needs to be responsive to recent performance without being destroyed by a single bad interaction. An exponential moving average (EMA) with α=0.2 is the right tool: it weights recent observations more heavily, forgets old ones gradually, and — as shown in Appendix B, Theorem B.4 — is Kalman-optimal for a noise ratio ρ=0.05. Both efficacy and confidence use the same EMA update rate.

agent/utility_scorer.py — EMA updates for E and C view on GitHub ↗ 
EFFICACY_ALPHA  = 0.2   # EMA learning rate for efficacy
CONFIDENCE_ALPHA = 0.2  # EMA learning rate for confidence

def _update_efficacy_ema(self, state: DomainState, efficacy_raw: float) -> float:
    """EMA-accumulated domain-level efficacy."""
    state.efficacy_ema = (
        (1 - self.EFFICACY_ALPHA) * state.efficacy_ema +
        self.EFFICACY_ALPHA * efficacy_raw
    )
    state.efficacy_ema = max(0.0, min(1.0, state.efficacy_ema))
    return state.efficacy_ema

def _update_confidence(self, state, test_pass_rate, contradiction_penalty, config):
    # The penalty is scaled by the field's penalty multiplier
    effective_penalty = contradiction_penalty * config.penalty_multiplier
    penalized_signal  = test_pass_rate * (1.0 - effective_penalty)

    state.confidence = (
        (1 - self.CONFIDENCE_ALPHA) * state.confidence +
        self.CONFIDENCE_ALPHA * penalized_signal
    )
    state.confidence = max(0.0, min(1.0, state.confidence))
    return state.confidence

The hard gate

When confidence or efficacy fall below the field minimum, the agent does not produce a low-quality answer. It produces no answer and escalates:

Dynamic difficulty routing

As domain confidence rises, the scorer signals that harder problems should be routed in — resetting the novelty counter and re-engaging the curiosity dynamics. This is visible in the simulation output where cycle 3 shifts to "medium" problems:

8. Putting it together: agent.run()

The UtilityAgent.run() method in agent/agent.py is where all components wire together. It processes one interaction end-to-end through 8 steps:

1
Field classification — FieldClassifier.classify(problem) returns a probability distribution over fields. get_effective_config() blends the config. High-stakes fields are floored at 15% minimum probability.
2
Contradiction detection — ContradictionDetector.check() runs logical, mathematical, and cross-session checks on the solution. Returns a ContradictionResult with penalties.
3
Arbiter (if competing solution provided) — ArbiterAgent.arbitrate() runs the 4-check pipeline, determines the verdict case (1–4), and if Case 3 (both wrong), opens a curiosity gap bonus on the subject.
4
Utility scoring — UtilityScorer.score() computes E_ema, C (with contradiction penalty), K_effective (with gap bonus), and final U. Sets below_minimum flag.
5
DPO pair collection — if contradictions were found, _record_dpo_pair() stores a (rejected=solution, weight=penalty_multiplier) pair. The preferred response is filled in later from the Arbiter's verified claim or human review.
6
Abstain gate — if task_score.below_minimum, the response is replaced with a formal abstention message. The raw answer is not returned. The caller should escalate.
7
Personality evolution — every 3 interactions (configurable), PersonalityManager.evolve() updates the behavioral wrapper based on the utility trend and contradiction rate in the domain.
8
Build and return AgentResponse — returns utility score, verdict, personality state, domain summary, active corrections, abstain flag, and recommended difficulty. Everything needed for monitoring and routing decisions.

What gets injected into the system prompt

agent/agent.py — get_system_prompt() view on GitHub ↗ 
def get_system_prompt(self, field: str) -> str:
    config = FIELD_CONFIGS.get(field, FIELD_CONFIGS["general"])
    wrapper_text = self.personality_manager.build_wrapper_prompt(field)

    corrections_block = ""
    if self.active_corrections:
        corrections_str = "\n".join(f"  - {c}" for c in self.active_corrections[-10:])
        corrections_block = f"\nACTIVE CORRECTIONS (verified — do not repeat these errors):\n{corrections_str}\n"

    return (
        f"You are an AI assistant operating in the {field} domain.\n\n"
        f"Minimum confidence standard for this field: {config.c_min:.0%}\n"
        f"If your confidence is below this threshold, you must abstain and recommend "
        f"consultation with a qualified professional. Do not guess.\n"
        f"{corrections_block}"      # ← session corrections injected here
        f"{wrapper_text}"           # ← personality wrapper (only if non-neutral)
        f"Always: state your reasoning, acknowledge uncertainty, never fabricate facts."
    )

The personality wrapper only injects when a trait deviates more than τ=0.05 from neutral. At the field-neutral point (after reset or on a new model release), the wrapper contributes nothing — it is identity-transparent by design.

9. The calibration pipeline

Why session-level corrections aren't enough

Injecting corrections into the system prompt fixes the current session. It doesn't fix the model. Next session, the same problem will resurface and need the same correction again. The calibration pipeline is what actually changes the weights — making the fix permanent and generalizing it to similar problems the model has never seen.

Why DPO, not RLHF

Direct Preference Optimization (DPO, Rafailov et al. 2023) is the right choice here for a specific reason: the utility function already generates exactly the data format DPO needs — (preferred, rejected) pairs with implicit reward. RLHF requires a separate reward model trained separately, adding a component that this system doesn't need. Every scored interaction where a contradiction is detected immediately produces a DPO pair.

agent/agent.py — DPO pair collection view on GitHub ↗ 
def _record_dpo_pair(self, task_id, field, preferred, rejected, weight, reason):
    self.calibration_dpo_pairs.append({
        "task_id":   task_id,
        "field":     field,
        "preferred": preferred,   # correct version (from Arbiter or human review)
        "rejected":  rejected,    # the contradictory output
        "weight":    weight,      # = field_config.penalty_multiplier → DPO loss weight
        "reason":    reason,
    })

def export_dpo_pairs(self, min_weight: float = 1.0) -> List[dict]:
    """Return accumulated pairs for the calibration pipeline."""
    return [p for p in self.calibration_dpo_pairs if p["weight"] >= min_weight]

The full calibration cycle

1. Collect all interactions since last run
2. Filter: keep pairs where U_preferred > U_rejected + threshold
   (avoid training on marginal differences — only clear signal)
3. Weight each pair by field_penalty_multiplier:
      DPO loss for surgery contradiction     → 10× weight
      DPO loss for software contradiction    →  2× weight
      DPO loss for creative writing mistake  →  1× weight
4. Mix with replay buffer (prior calibration pairs, weight=0.5)
   ← DO NOT SKIP THIS STEP (see §14)
5. Mix with golden examples (agent's own best responses, weight=0.7)
6. Run LoRA fine-tuning on weighted (preferred, rejected) pairs
7. Evaluate on held-out benchmark set
   → if benchmark U regresses, reject adapter and investigate
8. Deploy updated LoRA adapter if benchmark passes

What the replay buffer does. Without it, every calibration run erases what was fixed in the previous run. The model learns to not-do the new contradiction while re-learning how to do the old one. This is catastrophic forgetting at the calibration level — not the training level. The replay buffer mixes prior pairs with a down-weight (0.5) to prevent this. It is the most commonly skipped step and the most reliably regretted one.

Current status. The LoRA training step itself is not in this codebase — it requires a GPU and a base model checkpoint. The codebase produces the DPO pairs (export_dpo_pairs()) and validates that they accumulate correctly across cycles. The training step connects to standard HuggingFace trl DPO trainer with the pair format above. Full integration is Phase 6 of the roadmap.

10. Routing and the field classifier

Why routing matters so much

The routing quality experiment (four-arm controlled study, see §10.9.4) shows that routing is not a nice-to-have:

ArmCorrectnessΔ vs baselinep-value
A — No routing (generic prompt)59.0%——
B — Matched routing (oracle)71.5%+12.5%0.009
C — Mismatched routing (wrong specialist)41.5%−17.5%<0.001
D — VCG arbitration69.5%+10.5%0.029

Arm C is the critical result. A specialist fine-tuned on one domain is confidently wrong on out-of-domain queries — not just uncertain, but wrong with high confidence (Brier 0.292 vs 0.160). Misrouting is actively worse than no routing at all. This is the "Regime 2" failure mode, and it is why confidence-gated routing — not just domain classification — is required.

Why a discriminative model beats an LLM for classification

The field classifier in Phase 1 uses an LLM with keyword fallback. This is the weakest possible choice for routing, and it is explicitly noted in the codebase. Xu et al. (2024) show LLMs force-select a label even when none applies. Raval et al. (2026) show traditional ML models (fine-tuned DeBERTa, LightGBM) consistently outperform LLMs on structured classification. The planned upgrade (M1 mitigation, §10.4.1) replaces the LLM classifier with a discriminative model and retains the LLM only as a fallback for out-of-distribution queries.

11. The Arbiter Agent

Why it exists

In Phase 2, multiple specialist models may produce conflicting outputs on the same query. The Arbiter is the dedicated contradiction resolution agent that runs structured evidence checks — not a heuristic vote — and determines which output is correct, producing a formal verdict and internal evidence chain. Its design ensures that the "loudest" or most recently calibrated specialist cannot simply override others.

The four checks in order of cost

CHECK_WEIGHTS: Dict[str, float] = {
    "logical":       0.30,   # does output contradict its own premises?
    "mathematical":  0.40,   # are complexity/numerical claims provably wrong?
    "cross_session": 0.20,   # contradicts prior verified assertions?
    "empirical":     0.10,   # contradicts external ground truth? (stub — Phase 1 item)
}

VERDICT_CONFIDENCE_THRESHOLD = 0.85   # below this → Case 4 (escalate)

Checks run in this order because logical and mathematical checks are cheap (AST parsing, pattern matching) while cross-session requires a store lookup and empirical requires an external call. Arbiter confidence and case determination

agent/arbiter.py — _compute_confidence() and _determine_case() view on GitHub ↗ 
def _compute_confidence(self, checks: List[CheckResult]) -> float:
    """conf_arbiter = Σ(w_i × 1[check_i converges]) / Σ(w_i)"""
    numerator   = sum(CHECK_WEIGHTS[c.check_type] * (1.0 if c.converged else 0.0)
                      for c in checks)
    denominator = sum(CHECK_WEIGHTS[c.check_type] for c in checks)
    return numerator / denominator

def _determine_case(self, checks, arbiter_conf):
    if arbiter_conf < VERDICT_CONFIDENCE_THRESHOLD:
        return VerdictCase.CASE_4, None   # inconclusive → escalate

    converged = [c for c in checks if c.converged]
    # Weighted majority vote among converged checks
    vote_A = sum(CHECK_WEIGHTS[c.check_type] for c in converged if c.winner == "A")
    vote_B = sum(CHECK_WEIGHTS[c.check_type] for c in converged if c.winner == "B")
    vote_N = sum(CHECK_WEIGHTS[c.check_type] for c in converged if c.winner == "neither")

    if vote_N > vote_A and vote_N > vote_B: return VerdictCase.CASE_3, "neither"
    if vote_A > vote_B:                     return VerdictCase.CASE_1, "A"
    if vote_B > vote_A:                     return VerdictCase.CASE_2, "B"
    return VerdictCase.CASE_4, None

Calibration sampling — adaptive rate

agent/arbiter.py — adaptive sampling view on GitHub ↗ 
BASE_SAMPLE_RATE  = 0.03    # 3% of verdicts independently verified by default
SAMPLE_RATE_CEILING = 0.15  # hard cap — never exceeded

ADAPTIVE_SAMPLE_RATES = [
    (2.0, 0.10),   # correction_rate > 2× baseline → 10%
    (1.5, 0.08),   # correction_rate > 1.5× baseline → 8%
    (1.0, BASE_SAMPLE_RATE),
]

def adaptive_sample_rate(self) -> float:
    if self.total_verdicts < 20:
        return BASE_SAMPLE_RATE   # not enough data yet
    rate_ratio = self.correction_rate() / max(self._baseline_correction_rate, 0.01)
    for threshold, sample_rate in sorted(ADAPTIVE_SAMPLE_RATES, reverse=True):
        if rate_ratio >= threshold:
            return min(sample_rate, SAMPLE_RATE_CEILING)
    return BASE_SAMPLE_RATE

The adaptive sampling rate rises when the Arbiter is issuing many corrections — a signal that the specialists are drifting or the domain is changing. The 15% ceiling prevents runaway expert-sampling costs. For the VCG mechanism that makes this calibration continuous and self-correcting rather than sampled, see §10.6 of the whitepaper.

12. Blue-green deployment

Why it exists

In Phase 2, each specialist has its own independent calibration cycle and its own LoRA adapter. When a new adapter is ready, it cannot be deployed with a hard cutover — a regression in one specialist's new weights could harm thousands of live queries before it's detected. Blue-green deployment solves this by running the old model (BLUE) and the new candidate (GREEN) in parallel, shifting traffic gradually as measured utility confirms the new model is better.

Trigger condition

Trigger a deployment cycle when ALL of:
    |U_current - U_baseline| > δ(field)       ← significant deviation
    deviation sustained for ≥ T interactions   ← not transient noise
    held-out benchmark available               ← can evaluate candidate

δ(field) = base_δ / penalty_multiplier(field)    base_δ = 0.05

Surgery:          δ = 0.005   T ≥ 246 interactions  (very sensitive)
Software Eng:     δ = 0.025   T ≥  10 interactions  (responsive)
Creative Writing: δ = 0.050   T ≥   2 interactions  (fast)

The T values come from a power analysis on observed utility score variance (σ ≈ 0.04). High-stakes fields require large observation windows before any traffic shifts — a wrong conclusion about a surgery specialist is too costly to risk on thin data.

Traffic splitting via softmax

Phase 2 — Canary:  5% GREEN / 95% BLUE (minimum T_min interactions)

Phase 3 — Gradual shift:
    traffic_green = softmax(U_green, U_blue, τ=τ_field)
                  = exp(U_green/τ) / (exp(U_green/τ) + exp(U_blue/τ))

    Enforced floor/ceiling: clip(traffic_green, 0.05, 0.95)

    τ(field):
        Surgery:      τ = 0.05  → slow, winner-takes-all only when clear
        Software Eng: τ = 0.20  → moderate
        Creative:     τ = 0.50  → fast, follows any lead

Phase 4 — Promotion:  traffic_green ≥ promotion_threshold → 100%
Phase 5 — Retirement: GREEN holds through cooldown → BLUE weights freed

Rollback

Rollback triggers immediately when:
    U_green < U_blue - ε  for M interactions
    OR contradiction_rate_green > contradiction_rate_blue × 1.5
    OR benchmark regression > field_tolerance

Action: traffic_blue → 100% instantly
        GREEN failure cases added to replay buffer with negative weight
        Next candidate is trained against these failure modes

See §10.7–10.8 of the whitepaper for the full lifecycle specification and traffic split visualization.

14. What not to build first

These are the most common mistakes. All of them make Phase 1 harder without making it better.
  • Do not build a knowledge graph. The assertions store with embedding-based retrieval is sufficient for cross-session contradiction detection. A knowledge graph adds significant engineering complexity — schema design, graph queries, entity resolution — without improving the correction loop. The framework explicitly does not require one.
  • Do not build routing before the correction loop works. Routing multiplies whatever behavior the correction loop produces. If the correction loop isn't working correctly on a single model, routing will distribute broken behavior to multiple specialists instead of fixing it.
  • Do not skip the replay buffer. Every calibration run that omits it will silently erase prior corrections. The first time you see a previously-fixed error reappear, you will add the replay buffer under pressure. Add it at the start. It is one extra parameter in the training batch.
  • Do not lower C_min and E_min to make the system appear more capable. These are safety properties, not performance parameters. Lowering them means the system will produce confident wrong answers in domains where being wrong is dangerous. The simulation's abstain behavior is the correct behavior — the goal is to expand the domain where the agent can act confidently, not to suppress the abstention signal.
  • Do not build the empirical check (Arbiter stage 4) in Phase 1. It is expensive, introduces external dependencies, and is the least reliable check type. Logical and mathematical checks catch the most impactful contradictions at near-zero cost. Add empirical checks only after stages 1–3 are working correctly in production.
  • Do not treat the personality wrapper as permanent infrastructure. It is an interim behavioral mechanism designed to be superseded when fast specialist retraining makes it unnecessary. Do not build deep application logic that depends on specific personality trait values — the wrapper resets on every new model release by design.
